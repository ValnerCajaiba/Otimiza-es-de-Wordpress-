Dashboard de Otimização de Performance

Introdução

Este trabalho visa otimizar a velocidade de carregamento das páginas WordPress para reduzir o Connect Rate, diminuir o custo por visualização e aumentar o volume de vendas. Performance é experiência do usuário: página rápida = menos abandono, maior confiança e mais conversões.

As otimizações estão organizadas em 4 bases:

  • Boas práticas de desenvolvimento de páginas
  • Testes de velocidade;
  • Otimização de carregamento (fonts/BG, pré-carregamento, CSS, JS, imagens, heartbeat, FOIT/FOUT);
  • CDN e cache (Cloudflare + WP Rocket).

Regra de ouro: cada mudança deve ser testada. Performance só vale se não quebrar UX.

Boas práticas de desenvolvimento (WordPress + Elementor)

Plugins e Tema

Problema: Cada plugin injeta código na página (mesmo sem uso direto), aumentando peso, chance de conflitos e quebra de CSS.

Solução: Instalar o mínimo possível e desinstalar o que não for necessário. Atualizar plugins/tema com recorrência. Usar tema leve: Hello Elementor (recomendado).

Como implementar: Audite os plugins e remova o que não agrega. Após cada instalação/atualização, teste as páginas. Padrão da operação: Hello Elementor como base.

Menos widgets, mais semântica

Problema: Excesso de widgets para variações simples (ex.: tamanhos de texto) gera requisições, marcação suja e manutenção difícil.

Solução: Concentrar o conteúdo em um widget HTML usando classes CSS para variações.

Como implementar: Jeito certo: um widget HTML + CSS centralizado com classes (tipografia, espaçamentos, cores). Jeito errado: vários widgets só para alterar tamanho/estilo.

Evitar CSS inline

Problema: CSS inline quebra semântica, dificulta acessibilidade e prejudica SEO.

Solução: HTML limpo + classes semânticas + CSS centralizado (Aparência » Personalizar » CSS adicional).

Como implementar: Remover style="..." das tags. Criar classes (.titulo, .sub, .btn) e estilizar no CSS adicional. Resultado: melhor performance e manutenção.

Imagens (compactação, formato e tamanho)

Problema: Imagens são o maior vilão do carregamento: peso alto e dimensões desnecessárias.

Solução: Usar dimensão exata de renderização. Peso alvo: ≤ 100 KB (ideal). > 300 KB é inaceitável. Formatos: WebP (fotos/ilustrações) e SVG (ícones/vetores).

Como implementar: Compactar: tinypng.com. Redimensionar: iloveimg.com/pt/redimensionar-imagem. Converter em WebP: convertio.co. Subir já otimizadas (desktop e mobile).

Fontes (e nossa política operacional)

Problema: Cada tipo/peso de fonte = uma requisição. Fontes externas atrasam o carregamento.

Solução (padrão do mercado): Usar no máximo 2–3 pesos/tipos. Preferir WOFF2. Forçar font-display: swap (evita FOIT; permite FOUT controlado).

Particularidade da operação: Não subimos fontes localmente no WordPress por conta da clonagem para múltiplos domínios. Isso gera requisições cruzadas e problemas de path. Em vez disso, ativamos o “Auto-hospedar fontes Google” do WP Rocket, que puxa e serve as fontes otimizada e automaticamente por domínio.

Como otimizar fontes do Google (nosso padrão): WP Rocket → Otimização de arquivos → ativar “Otimizar entrega de fontes” / “Auto-hospedar fontes Google”. Garantir font-display: swap via snippet (abaixo). Reduzir pesos: só o que realmente usa (ex.: 400/700).

Snippet (Code Snippets) – garantir swap no Elementor Pro:

add_filter( 'elementor_pro/custom_fonts/font_display', function( $current_value, $font_family, $data ) {
  return 'swap';
}, 10, 3 );

Se algum projeto não for clonado para múltiplos domínios, opção alternativa é subir WOFF2 em Elementor Pro » Custom Fonts. Para a nossa operação, mantemos o padrão WP Rocket.

Evitar requisições externas

Problema: Cada recurso em servidor externo adiciona latência e incerteza.

Solução: Hospedar imagens e fontes no mesmo domínio (respeitando nossa política de fontes via WP Rocket) e minimizar chamadas de terceiros.

Como implementar: Trocar URLs de imagens externas por uploads locais. Evitar bibliotecas externas quando houver alternativa local.

Testes de velocidade

Ferramentas usadas

  • PageSpeed Insights
  • GTmetrix
  • Lighthouse (Chrome DevTools)
  • WebPageTest (Complementar)

Métricas principais (o que acompanhar)

  • FCP: quando aparece o primeiro conteúdo.
  • LCP (< 2,5s): maior elemento visível (imagem/bloco principal).
  • TBT (< 200ms): tempo bloqueado por scripts longos.
  • CLS (< 0,1): estabilidade do layout.
  • Speed Index (< 3s): rapidez de preenchimento visual.
  • TTI (< 5s): tempo até ficar totalmente interativa.
  • Page Size: (quanto menor, melhor).
  • Requests: (< 60 na primeira dobra é uma boa meta).

Dica: pré-carregar o primeiro elemento (ex.: imagem LCP) pode ajudar FCP/LCP.

Comparação: WebPageTest vs GTmetrix vs PageSpeed Insights

Critério / característica WebPageTest GTmetrix PageSpeed Insights
Profundidade de análise Muito detalhado (filmstrip, waterfall, request-by-request) Detalhado, ótimos gráficos e auditorias Menos detalhado por requisição, foco em Web Vitals
Customização de teste Alta (local, rede, navegador) Boa (locais, dispositivos, rede) Baixa (configurações limitadas)
Medição com navegador real Sim Sim Ambiente controlado + dados de campo
Histórico/monitoramento Básico no free; melhor no pago Sim (histórico e comparações) Não (teste pontual)
Alinhamento com Google/SEO Útil, mas não “oficial” Integra Lighthouse/Web Vitals Forte alinhamento com foco em Web Vitals
Facilidade p/ iniciantes Complexo Médio Muito fácil
Uso gratuito / limites Free com filas/limites Free com restrições Totalmente gratuito
Variação dos resultados Pode variar por infraestrutura Pode variar por local/rede Mais constante, mas pode divergir da realidade

Em resumo (prático):

  • Use WebPageTest quando precisar investigar a fundo gargalos (filmstrip/waterfall).
  • Use GTmetrix para visão equilibrada (gráficos, histórico, comparações).
  • Use PageSpeed Insights para ver como o Google “enxerga” seu site e priorizar Web Vitals.

É comum combinar: PageSpeed (prioridades) → GTmetrix/WebPageTest (diagnóstico detalhado).

Plano de Otimização – WordPress + Elementor + WP Rocket + Code Snippets

Cache e Regras Avançadas (WP Rocket + Cloudflare)

Problema: Sem regras claras de cache, páginas dinâmicas quebram (formulários, checkout, login) ou ficam lentas por armazenar dados desnecessários. Por outro lado, páginas estáticas sem cache deixam de aproveitar a performance máxima.

Solução: Combinar regras no Cloudflare e no WP Rocket para equilibrar cache agressivo em páginas estáticas e exceções em páginas dinâmicas.

Como implementar:

Cloudflare

  • Criar regra “Sem cache wp-admin”: Condição: URI Path contém /wp-admin → ação: Bypass cache.
  • Criar regra “Cache Everything” para o domínio: Edge TTL alto em páginas estáticas (até 6 meses). Browser TTL menor em páginas dinâmicas (algumas horas/dias).

Ordem importa → a regra do wp-admin deve vir primeiro. Se precisar diferenciar slugs específicos (ex.: página de obrigado, políticas), crie regras adicionais.

WP Rocket

  • Tempo global de cache: 2–4h (nossas páginas mudam constantemente).
  • Excluir URLs estáticas de baixa atualização: ex.: /gracias/, /termos/.
  • Regras avançadas → “URL(s) Jamais em Cache”: adicionar slugs específicos.

Prós e Contras: Cache agressivo melhora bastante performance, mas pode causar inconsistências em páginas com atualização constante. Cache curto resolve inconsistência, mas aumenta requisições ao servidor. O equilíbrio depende do tipo de página.

Delay / Defer de JavaScript

Problema: Scripts bloqueiam renderização (TBT alto). Porém, atrasar todos os JS pode quebrar funcionalidades críticas (VSL, chatbot, tracking).

Solução: Atrasar ou adiar apenas scripts não críticos, mantendo os essenciais intactos.

Como implementar (WP Rocket): Ativar “Atrasar execução de JS”. Inserir exceções: Scripts de VSL (VTurb). Scripts de tracking (Pixel, Google Tag, etc.). Chatbots. Ativar “Carregar JavaScript adiado (defer)” para o restante.

Particularidades da operação: Em páginas de VSL, nunca atrasar JS → o player trava. Em páginas com chatbot, atrasar JS gera lentidão no envio/recebimento de mensagens. Lista de exclusões deve ser mantida como checklist vivo.

Prós e Contras: Prós: melhora grande em métricas de bloqueio. Contras: alto risco de quebra se mal configurado → exige testes cuidadosos.

LazyLoad de imagens / iframes / vídeos

Problema: Imagens e vídeos pesados aumentam LCP e carregam desnecessariamente antes do usuário rolar a página.

Solução: Carregar imagens e vídeos sob demanda, mas excluir os elementos da primeira dobra (logo, herói, imagem LCP).

Como implementar (WP Rocket): Ativar LazyLoad para imagens, iframes e vídeos. Ativar substituição do iframe do YouTube pela thumb (carrega só 10 KB até clicar em play, em vez de 1 MB). Excluir do LazyLoad (campo de exclusão do WP Rocket): .header-logo, .hero .elementor-image img, .slider-hero img.

Prós e Contras: Prós: páginas longas ficam muito mais leves. Contras: se esquecer de excluir imagens críticas (LCP), o efeito pode ser contrário → aumento de métricas.

CSS (otimização de entrega)

Problema: CSS bloqueante atrasa renderização, e excesso de código inútil aumenta o tamanho do arquivo.

Solução: Minificar, remover CSS não usado e entregar CSS crítico primeiro.

Como implementar (WP Rocket + Elementor): WP Rocket → Ativar “Minificar CSS”. WP Rocket → Ativar “Otimizar entrega de CSS”. Elementor → Ferramentas → “Limpar arquivos e dados”.

Prós e Contras: Prós: menos peso, renderização mais rápida. Contras: pode quebrar estilos específicos → sempre validar visualmente após a limpeza.

Fontes e Backgrounds

Problema: Fontes externas e backgrounds pesados aumentam requisições e carregamento.

Solução: Usar auto-hospedar fontes Google (WP Rocket). Ativar font-display: swap. Converter imagens de fundo para WebP ou substituir por gradientes CSS.

Como implementar: Elementor Pro → Custom Fonts (quando não há clonagem de domínios). Nossa operação → WP Rocket “Auto-hospedar fontes Google”. Code Snippets → adicionar font-display: swap. Converter backgrounds > 100 KB em WebP. Sempre que possível, substituir BGs por gradientes CSS.

Prós e Contras: Prós: fontes rápidas, BGs leves, experiência mais fluida. Contras: ao usar fontes locais em clones → quebra em múltiplos domínios (por isso não adotamos).

Pré-carregamento (Preload / Preconnect)

Problema: Recursos críticos (LCP, fontes) demoram a ser servidos sem preload.

Solução: Pré-carregar apenas recursos críticos e inevitáveis.

Como implementar (WP Rocket): Em “Pré-carregamento”, adicionar fontes WOFF2 usadas no topo (ex.: Roboto-700.woff2). Evitar excesso (só o que for crítico). Pré-carregar a imagem LCP quando fizer sentido.

Prós e Contras: Prós: melhora LCP e FCP. Contras: excesso de preloads pode até piorar a performance.

Heartbeat

Problema: Heartbeat consome recursos do servidor sem necessidade no front-end.

Solução: Reduzir no painel e edição, desativar no front.

Como implementar (WP Rocket): Painel/Admin: “Reduzir atividade”. Edição: “Reduzir atividade”. Front-end: “Desativar” (se não houver edição ao vivo).

Garantir texto antes da fonte (FOIT → FOUT controlado)

Problema: Sem configuração, o usuário vê “tela branca” até a fonte carregar (FOIT).

Solução: Trocar FOIT por FOUT: texto aparece com fonte fallback e depois troca para a fonte final.

Como implementar (Code Snippets):

add_filter( 'elementor_pro/custom_fonts/font_display', function( $current_value, $font_family, $data ) {
  return 'swap';
}, 10, 3 );

Testes, validação e rollback

Problema: Sem validação, é impossível saber se as otimizações funcionaram ou quebraram a página.

Solução: Testar em várias ferramentas + navegação real + rollback rápido.

Como implementar: Limpar caches (WP Rocket + CDN). Regenerar CSS/Dados (Elementor). Testar em abas anônimas. Validar funções críticas: menus, checkout, reCAPTCHA, tracking. Medir novamente em PageSpeed, WebPageTest, GTmetrix. Documentar “Antes” e “Depois”.

Rollback

  • Desativar Delay JS.
  • Remover exclusões agressivas.
  • Desativar otimização de CSS.
  • Remover snippets recentes.
  • Restaurar backup se necessário.

Ordem recomendada de implementação

  1. Boas práticas de desenvolvimento.
  2. Imagens (WebP + redimensionadas).
  3. Fontes (Google auto-hospedadas + swap).
  4. CSS (otimizar entrega).
  5. JS (Delay/Defer + exceções).
  6. LazyLoad (excluir primeira dobra).
  7. Heartbeat.
  8. CDN sem cookies (quando aplicável).
  9. Testes + métricas + rollback.

Particularidades da nossa operação

  • Clonagem de domínios: evitar fontes locais → usar auto-hospedar do WP Rocket.
  • SEO Google: imagens precisam estar no domínio → não usamos subdomínio CDN para assets.
  • VSL (VTurb): embed exige remover Delay JS.
  • Chatbot: se JS atrasado → tempo de resposta lento.

Conclusão

Aplicando este manual, já tivemos ganhos significativos em velocidade e estabilidade das páginas. O próximo passo é evoluir também em layout e experiência do usuário, garantindo que a performance técnica esteja sempre alinhada à eficácia comercial.

Comparativo de Plataformas para Páginas Estáticas

Característica HTML, CSS & JavaScript Puro WordPress (com Elementor) Atomicat

Performance Potencialmente a mais rápida, com código mínimo e otimizado. Pode ser rápida com otimização (WP Rocket, etc.), mas plugins e tema adicionam peso. Boa performance, otimizado para conversão.
